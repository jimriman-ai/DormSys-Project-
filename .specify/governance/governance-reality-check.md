# Governance Reality Check
Status: Operational Control Procedure
Purpose: Determine actual governance state using canonical evidence only
Scope: Any spec, program, or execution stream where lifecycle ambiguity exists

## 1. Intent

This procedure determines the real governance position of work.
It exists to stop stage fabrication, authority inference, false blockers, and governance drift before remediation begins.

This procedure does not redesign process.
This procedure does not create rules.
This procedure does not propose new lifecycle stages.

It only answers:
- what is completed,
- what is active,
- what is authorized next,
- what is not grounded,
- and what must not be assumed.

## 2. Input Priority

Use evidence in this order only:

1. Canonical authority records
2. Canonical lifecycle/governance documents
3. Explicit governance records tied to the authority model
4. Supporting documents that mirror but do not own decisions
5. Descriptive artifacts for context only

The following may not be used to create authority:
- `spec.md`
- `plan.md`
- `tasks.md`
- status headers
- progress notes
- AI summaries
- review prose without canonical gate definition

## 3. Mandatory Questions

The reality check must answer all of the following.

### Q1. What is the last officially completed stage?
Only report a stage if canonical evidence supports completion.

### Q2. What is the current active stage?
Only report a stage if canonical evidence supports active status.

### Q3. What is the next authorized step?
Report only the next step supported by canonical governance.

### Q4. What labels are being used informally?
List all non-canonical labels, invented stages, derived readiness terms, or convenience names.

### Q5. What authority source governs the next decision?
Identify exactly one canonical authority source.

### Q6. Are there blockers or partial constraints?
If yes, list only blockers with explicit record support.

### Q7. What claims are not grounded?
List all conclusions that cannot be supported by canonical evidence.

### Reality Check Record

- Final Verdict:

## 5. Final Verdict Rules

The final verdict must end in one of these outcomes:

- `Grounded and Clear`
- `Grounded but Blocked`
- `Partially Grounded`
- `Not Grounded`
- `Authority Conflict - Halt`
- `Lifecycle Ambiguity - Halt`

## 6. Prohibited Actions During Reality Check

Do not:
- invent a new governance phase,
- create an unofficial review gate,
- upgrade an observation into authority,
- infer permission from planning artifacts,
- infer scope from task breakdown,
- classify an undefined dependency as blocking without explicit record,
- propose remediation before state is grounded.
